Module 10 · APIs

Two computers talking politely.

An API is just a doorway between programs — one asks, the other answers, both follow a set of manners. Once you see the four polite words (GET, POST, PUT, DELETE), the locked door (auth), and the queue rules (rate limits), every API in the world looks the same.

The Case · Trailblaze Logistics

Building the live shipment dashboard for a nationwide courier

Trailblaze moves 80,000 shipments a day across 280 cities. Their operations team needs a dashboard that pulls real-time package status, lets them update delivery attempts, and flags problems. Every lesson here talks to one fictional REST API: api.trailblaze.in/v1. By the end you'll know exactly how that conversation works — request by request.

1HTTP methods · the four polite verbs

GET, POST, PUT, DELETE — and what each one actually does

Every API request is a verb plus a noun: "GET this shipment" or "DELETE that user". There are dozens of HTTP methods in theory, but in practice you'll use four. Each does one job. Click each verb to see exactly what happens between Trailblaze's app and the API.

Think of it like a librarian. GET = "show me book #42" (read, no change). POST = "here's a new book, add it" (create). PUT = "replace the details of book #42 with these" (update). DELETE = "remove book #42 from the shelf". The URL points to the resource; the verb says what to do with it.

import requests

base = "https://api.trailblaze.in/v1"

# GET — fetch a single shipment
r = requests.get(f"{base}/shipments/TBZ-48291")
shipment = r.json()

# POST — create a new shipment
new_ship = {"origin": "Mumbai", "dest": "Pune", "weight_kg": 2.3}
r = requests.post(f"{base}/shipments", json=new_ship)

# PUT — replace the delivery address
update = {"dest_pincode": "411014", "phone": "+91-9XXXXX1234"}
r = requests.put(f"{base}/shipments/TBZ-48291", json=update)

# DELETE — cancel the shipment
r = requests.delete(f"{base}/shipments/TBZ-48291")
assert r.status_code == 204  # 204 = No Content, success

The hotel-room mnemonic

Imagine a hotel front desk. GET = "let me see room 304" (no key, just the registry). POST = "I'd like to book a room please" (you don't pick the room number — the hotel assigns it). PUT = "I'm in 304 and I want the king bed swapped for twin beds" (replace details of an existing room). DELETE = "cancel my booking". If you remember this, the whole REST world clicks.

2JSON parsing · reading what the API hands back

From a wall of text to a Python value you can use

APIs almost always reply in JSON — a nested format with keys, values, and lists. The trick is navigating to the specific value you want. Click any line in the response tree to see the exact Python path that gets you to that value.

JSON has just six building blocks: objects (curly braces with key-value pairs), arrays (square brackets with items), strings, numbers, booleans, and null. In Python, an object becomes a dict, an array becomes a list. Once you've called r.json(), you navigate it with the same square brackets you'd use on any dict or list.

import requests

r = requests.get("https://api.trailblaze.in/v1/shipments/TBZ-48291")
data = r.json()  # ← turns the JSON text into a Python dict

# Top-level fields are just dict keys
ship_id  = data["shipment_id"]
status   = data["status"]

# Nested objects — chain the brackets
sender   = data["sender"]["name"]
city     = data["recipient"]["address"]["city"]

# Arrays — index by position, then dig in
first_scan_at = data["events"][0]["timestamp"]
last_event    = data["events"][-1]["location"]

# Safer access for fields that might be missing
delivered_at  = data.get("delivered_at", None)
print(f"{ship_id}: {status}, last seen {last_event}")

The matryoshka rule

Every JSON response is a nested Russian doll. Start from the outermost layer and work inward, naming what you see: "It's a dict → with a key called events → that holds a list → whose first item is a dict → with a key called timestamp." Say that sentence in your head and the bracket chain data["events"][0]["timestamp"] writes itself.

3Authentication · API keys and Bearer tokens

Three ways the API decides whether to talk to you

Most useful APIs won't answer strangers. You have to identify yourself in a header that travels with every request. Switch between "no auth", "API key", and "Bearer token" to see the same request fail and then succeed — purely because of one line added to the headers.

An API key is a long string Trailblaze gives you when you sign up — like a club membership card. A Bearer token is the same idea but usually short-lived (it expires after a few hours), commonly issued after a separate login step. Both travel in HTTP headers, not the URL — never put secrets in the URL itself, because URLs end up in server logs and browser history.
Same request, three auth styles · see what changes
# curl-style view of the request GET /v1/shipments/TBZ-48291 HTTP/1.1 Host: api.trailblaze.in Accept: application/json --- response --- HTTP/1.1 401 Unauthorized Content-Type: application/json {"error": "Missing authentication credentials", "hint": "Add an Authorization header"}
⚠ Request rejected · 401 Unauthorized · the server doesn't know who you are
What just happened? No headers identifying the caller, so the API turns you away at the door before doing any work. This is the default behavior for nearly every commercial API on the internet.
# Same request — but with an API key header GET /v1/shipments/TBZ-48291 HTTP/1.1 Host: api.trailblaze.in Accept: application/json X-Api-Key: tbz_live_4f8a92e6c1d3b7a0f5e8d4c2b9a7e3f1 --- response --- HTTP/1.1 200 OK Content-Type: application/json {"shipment_id": "TBZ-48291", "status": "In transit", "location": "Pune Hub"}
✓ 200 OK · the API recognized your key and returned the shipment
API keys are long, opaque strings tied to your account. They usually go in a custom header like X-Api-Key or as a query parameter (less secure). Treat them like passwords — never commit them to GitHub. We'll fix that properly in Module 14 with environment variables.
# OAuth-style Bearer token — usually short-lived GET /v1/shipments/TBZ-48291 HTTP/1.1 Host: api.trailblaze.in Accept: application/json Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... --- response --- HTTP/1.1 200 OK Content-Type: application/json {"shipment_id": "TBZ-48291", "status": "In transit", "location": "Pune Hub"}
✓ 200 OK · token accepted, valid for the next 60 minutes
Bearer tokens go in the standardized Authorization header with the literal word "Bearer" in front. They're typically issued by a login endpoint, expire quickly, and can be refreshed. Most modern OAuth-protected APIs (Google, GitHub, Spotify) use this pattern.
import requests
import os

# API key style — custom header
headers = {"X-Api-Key": os.environ["TRAILBLAZE_KEY"]}
r = requests.get("https://api.trailblaze.in/v1/shipments/TBZ-48291",
                 headers=headers)

# Bearer token style — Authorization header
token = os.environ["TRAILBLAZE_TOKEN"]
headers = {"Authorization": f"Bearer {token}"}
r = requests.get("https://api.trailblaze.in/v1/shipments/TBZ-48291",
                 headers=headers)

# Reuse the headers across many calls — use a Session
s = requests.Session()
s.headers.update({"Authorization": f"Bearer {token}"})
r1 = s.get("https://api.trailblaze.in/v1/shipments/TBZ-48291")
r2 = s.get("https://api.trailblaze.in/v1/shipments/TBZ-48292")
# Both requests carry the token automatically

Read the error, not the silence

When an API call fails, the error code is doing real work. 401 = "I don't know who you are" (auth missing/wrong). 403 = "I know who you are, but you're not allowed" (permission issue). 404 = "that resource doesn't exist". 429 = "you're going too fast" (we'll meet this in Lesson 5). 5xx = "it's my fault, try again later". Always print r.status_code and r.text on failure — the API is telling you what's wrong.

4Pagination · 80,000 shipments, 50 at a time

Why APIs hand you data in slices — and how to walk through all of it

Trailblaze handles 80,000 shipments a day. If you asked the API "give me everything", the response would be a 200 MB JSON blob and the server would melt. APIs solve this by paginating — they return a small slice plus instructions on how to ask for the next slice. Page through the data below.

The two common pagination styles: offset-based (you say "give me 50 items, starting from item 100") with parameters like ?page=3&limit=50 or ?offset=100&limit=50; and cursor-based (the API hands you a token like next_cursor=abc123 and you pass it back to get the next chunk). Offset is simpler; cursor is more reliable when data is changing while you read.
Walk through 80,000 shipments · 50 per page
import requests

base = "https://api.trailblaze.in/v1/shipments"
headers = {"X-Api-Key": "tbz_live_..."}

# ─── Offset-style pagination ──────────────────────────────
all_ships, page = [], 1
while True:
    r = requests.get(base, headers=headers,
                     params={"page": page, "limit": 50})
    body = r.json()
    all_ships.extend(body["results"])
    if page >= body["total_pages"]: break
    page += 1

# ─── Cursor-style pagination (more robust) ────────────────
all_ships, cursor = [], None
while True:
    params = {"limit": 50}
    if cursor: params["cursor"] = cursor
    body = requests.get(base, headers=headers, params=params).json()
    all_ships.extend(body["results"])
    cursor = body.get("next_cursor")
    if not cursor: break

print(f"Pulled {len(all_ships):,} shipments")

The "first three pages only" rule

When you're exploring a new API, never write the full pagination loop on attempt one. First call the endpoint with page=1 and just print the response keys. You'll discover whether the data lives under results or data or items, how the total is reported, and whether pagination is offset or cursor. Only then wrap it in a loop. Many people debug a broken pagination loop for an hour before realizing the data was under a key they hadn't checked.

5Rate limiting · don't hammer the door

Why you'll see 429s — and how to respect the queue

Every API caps how often you can ask. Trailblaze allows 10 requests per minute on the free tier. Click the FETCH button rapidly and watch the quota burn, then the 429s start. Notice the Retry-After header — that's the API telling you exactly how long to wait.

Rate limiting protects the API from being overwhelmed and protects you from accidentally DDoS-ing someone. When you hit the limit you get 429 Too Many Requests with a Retry-After header (seconds to wait). The right reaction in code is exponential backoff — wait, retry, wait longer, retry, give up after N tries. Never hammer in a tight retry loop.

import requests
import time

def fetch_with_backoff(url, headers, max_tries=6):
    delay = 1
    for attempt in range(max_tries):
        r = requests.get(url, headers=headers)
        if r.status_code == 429:
            wait = int(r.headers.get("Retry-After", delay))
            print(f"Rate limited — sleeping {wait}s")
            time.sleep(wait)
            delay *= 2   # exponential backoff
            continue
        r.raise_for_status()
        return r.json()
    raise RuntimeError("Gave up after too many 429s")

# Also: stay UNDER the limit proactively
PER_MIN = 10
for ship_id in shipment_ids:
    fetch(ship_id)
    time.sleep(60 / PER_MIN)   # pace yourself: 6 sec between calls

The good citizen rule

Before scaling up any script that calls a third-party API, read the rate-limit section of their docs. Then deliberately run your script slower than the limit — if the cap is 60/min, pace yourself at 50/min. The extra headroom protects you when something else on your network briefly delays a call. The polite, predictable script that runs all night beats the aggressive one that gets your API key suspended at 3 AM.
