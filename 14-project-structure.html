From notebook chaos to a real, deployable project

Your churn analysis notebook is brilliant — but your manager can't run it, your laptop crashed last week and lost the API key, and nobody else on the team can reproduce your numbers. Time to grow up. This module shows how senior data scientists organise their code so it works on someone else's machine, today and three months from now.

Case Study

Turning your churn notebook into a project your manager can run

You're Arjun Patel, a data scientist at a fintech. You spent two weeks in a Jupyter notebook predicting churn — it works beautifully on your laptop. Your manager wants the analytics team to run it weekly. They use Windows, you use Mac. They don't have your API keys. They don't know which versions of pandas you used. Restructure the project so it just works.

1

The standard layout — click any file to see what goes in it

A 600-line notebook is impossible for anyone (including future-you) to navigate. A well-laid-out project splits responsibilities: data loading, processing, modelling, and orchestration each get their own home.

The folder you see below is the de facto standard for production Python data projects. It's used by teams at Spotify, Stripe, ISRO, and everywhere in between. The key idea: code that runs goes in src/, files that run code go in scripts/. Configuration, secrets, data and tests each live in their own folder so you can change one without breaking the others.

A great rule of thumb: if you ever write the same code in two notebooks, it belongs in src/utils.py. The moment you find yourself copy-pasting a function, refactor it into a module and import it from both places.
2

Never hardcode secrets — use .env files

An API key in your code is one screenshot away from being on Twitter. Compare the "amateur" version of data_loader.py with the "professional" one — same logic, vastly different security.

Environment variables are name=value pairs the operating system holds in memory. The python-dotenv library reads them from a file named .env when your script starts. The golden rule: the .env file never gets committed to git. Each team member keeps their own copy, and you ship a .env.example with placeholders to show the shape.
Toggle between the two approaches
# src/data_loader.py import os from dotenv import load_dotenv load_dotenv() # reads .env into os.environ API_KEY = os.environ["NSE_API_KEY"] # raises if missing — fail fast DB_URL = os.environ.get("DB_URL", "sqlite:///local.db") EMAIL_PWD = os.environ["EMAIL_APP_PASSWORD"] def fetch_churn_data(): headers = {"Authorization": f"Bearer {API_KEY}"} return requests.get("https://api.fintech.in/churn", headers=headers)
Always add .env to your .gitignore before your first commit. Also ship a .env.example file with empty values (e.g. NSE_API_KEY=) so teammates know exactly which variables they need to set on their machine.
3

Pin your dependencies — build a requirements.txt

Tap the packages you actually use — watch the requirements file build itself. Each line is a contract: "install this package at this version, no exceptions."

Without a requirements.txt, your project is a time bomb. Today pandas is version 2.2.1 — it might be 3.0 next year with breaking changes. Pinning means "==" for production code; ">=" only when you genuinely don't care which patch version. Running pip freeze dumps exactly what's installed in your virtual environment — copy-paste that into your file for full reproducibility.

# Set up a clean virtual environment $ python -m venv .venv $ source .venv/bin/activate # Mac/Linux $ .venv\Scripts\activate # Windows # Install everything from the pinned file $ pip install -r requirements.txt # Later, after you've added new packages: $ pip freeze > requirements.txt # Your teammate clones the repo, then: $ git clone https://github.com/yourorg/churn-project $ cd churn-project $ python -m venv .venv && source .venv/bin/activate $ pip install -r requirements.txt $ python scripts/run_weekly.py # works first try!
For really polished projects, look at Poetry or uv — they pin transitive dependencies too (the packages your packages depend on). But requirements.txt with pip freeze is the 80/20 starting point and works everywhere.
4

The README — your project's front door

A great README answers four questions in under 30 seconds: what does this do, how do I install it, how do I run it, who do I ask if something breaks? Click sections to add — preview renders live.

The README is the very first thing anyone — a new teammate, your manager, your future self — sees when they open your project on GitHub. A bad README means the project might as well not exist. Markdown (.md) is the universal language: GitHub, Jupyter, Notion, Obsidian and most IDEs render it consistently.
