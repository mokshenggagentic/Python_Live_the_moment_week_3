Storing Scraped & API Data
Module 12 · Storage

Pulled the data — now keep it.

Scraping or calling an API gives you data once. The interesting work happens when you keep doing it — every hour, every day — and the data starts piling up into a history you can actually analyze. Today we build that pile, carefully.

📈
The Case · DalalStreet Tracker

A personal data lake of NSE stock prices, grown one day at a time

You want to track the closing prices of 12 NSE-listed companies for your own research. No expensive Bloomberg terminal — just a Python script that runs every weekday after market close, scrapes the prices, and appends a clean row per stock to a CSV. Over months it becomes your personal historical dataset.

1Appending to CSV · the easiest persistent storage

One file, one row per fetch, growing forever

A CSV is just a text file with comma-separated rows. To save scraped data, you don't need a database — you open the file in append mode and write one row per record. Click "Run scraper" a few times and watch the file grow.

Python file modes that matter for storage: "w" overwrites the file (dangerous — destroys history), "a" appends to the end (safe — keeps history), "x" creates only if missing (errors if exists). For tabular data, use pandas: df.to_csv("file.csv", mode="a", header=not exists). The header= trick writes column names only on the first run.
Click Run repeatedly · each run appends a fresh batch of rows

import pandas as pd
from pathlib import Path
from datetime import datetime

CSV_PATH = Path("data/nse_prices.csv")
CSV_PATH.parent.mkdir(parents=True, exist_ok=True)

def scrape_prices():
    # ... fetch from NSE / Yahoo / your source ...
    return pd.DataFrame([
        {"date": "2026-05-25", "symbol": "RELIANCE", "close": 2812.40},
        {"date": "2026-05-25", "symbol": "TCS",      "close": 3654.10},
    ])

def append_to_csv(df, path):
    file_exists = path.exists()
    df.to_csv(
        path,
        mode="a",                          # append, don't overwrite
        header=not file_exists,            # write header only first time
        index=False
    )

prices = scrape_prices()
append_to_csv(prices, CSV_PATH)
print(f"Appended {len(prices)} rows at {datetime.now()}")

Always add the timestamp column yourself

The most common scraper bug: forgetting to record when each row was scraped. Your CSV has the closing price but no clue which trading day it belonged to, because the page just said "Today: ₹2812". Add a scraped_at or date column to every row before appending — your future self will thank you when those rows look identical six months later.

2Deduplication · stop the same row from showing up twice

Why scrapers always produce duplicates — and how to clean them

Your scraper might run twice by accident, or the source might re-publish the same record under a slightly different timestamp. Pick a key (the column or columns that should be unique), then drop duplicates by that key.

The key insight: "duplicate" depends on what you call the same. Two rows with identical symbol+date are duplicates if you only want one price per stock per day. The same two rows might NOT be duplicates if you also care about scrape_time. Pick your uniqueness key first, then use df.drop_duplicates(subset=[...], keep="last") to clean.

import pandas as pd

df = pd.read_csv("data/nse_prices.csv", parse_dates=["date"])

# ─── Drop rows that are duplicates across ALL columns ──────
df = df.drop_duplicates()

# ─── Better: dedupe by your business key ───────────────────
# "one row per symbol per day" → key is (symbol, date)
df = df.drop_duplicates(subset=["symbol", "date"], keep="last")

# keep="last" → if a later scrape corrected the price, keep that
# keep="first" → trust the first value you scraped, ignore later ones

# ─── Defensive pattern: dedupe BEFORE appending ────────────
def safe_append(new_df, path, key_cols):
    if path.exists():
        old = pd.read_csv(path)
        combined = pd.concat([old, new_df], ignore_index=True)
        combined = combined.drop_duplicates(subset=key_cols, keep="last")
        combined.to_csv(path, index=False)
    else:
        new_df.to_csv(path, index=False)

safe_append(prices, CSV_PATH, key_cols=["symbol", "date"])

Decide the uniqueness key on day one

Before you write a single line of storage code, answer this question on paper: "What makes two of my rows the same row?" For NSE prices: symbol + date. For tweets: tweet_id. For shipments: shipment_id. For weather: city + timestamp. Write the answer in a comment at the top of your file. Then everywhere you append, dedupe by that exact key. This single habit prevents the most common data quality issue in scraped datasets.

3The schedule library · cron without leaving Python

Tell jobs to run every X minutes, every day at Y o'clock

The schedule library is the simplest way to make a Python script run things on a timetable. Press Start to watch a sped-up day: a job firing every "5 minutes", another at "09:30 every weekday", and a daily cleanup at "midnight".

Built right into Python's mental model: schedule.every(5).minutes.do(scrape) reads like an English sentence. The library doesn't run your job — it just remembers when to. You sit in a while True loop that calls schedule.run_pending() every second, and the library fires the right jobs at the right times. Great for development and small servers; not great for production HA (use cron or systemd for that).

import schedule
import time

def scrape_intraday():    print("📊 quick price snapshot")
def scrape_open_prices(): print("📈 market opened — full scrape")
def dedupe_and_archive(): print("🧹 nightly cleanup")

# Read the schedule like English sentences
schedule.every(5).minutes.do(scrape_intraday)
schedule.every().monday.at("09:30").do(scrape_open_prices)
schedule.every().tuesday.at("09:30").do(scrape_open_prices)
# ...add Wed, Thu, Fri the same way (or use a loop)
schedule.every().day.at("00:00").do(dedupe_and_archive)

# The library just remembers what to fire when.
# YOU have to keep the loop running so it can fire jobs.
while True:
    schedule.run_pending()
    time.sleep(1)

Always log the start and end of each job

Wrap every scheduled function with print(f"START {name} at {datetime.now()}") and the same at the end. When something silently breaks at 3 AM, those logs are the only way you'll know which job ran, when, and whether it finished. Production teams use a logging module, but even print() redirected to a log file is enough for your personal scraper.

4Cron basics · the language servers actually speak

Five numbers that schedule the entire UNIX world

When you deploy your scraper to a real server, you won't keep a Python loop running 24/7 — you'll register the job with cron. Cron schedules look like 30 9 * * 1-5. Click a preset or read each field below.

A cron expression has 5 fields separated by spaces: minute hour day-of-month month day-of-week. Each field is either a number, a range (1-5), a list (1,15), a step (*/15), or * meaning "every". Once you can read one cron line, you can read every cron line ever written.

Use crontab.guru while you're learning

The website crontab.guru shows the English meaning of any cron expression as you type. Use it for the first dozen schedules you write — eventually the patterns become obvious. The five most useful expressions you'll see in real codebases: 0 * * * * (every hour), 0 0 * * * (every midnight), 0 9 * * 1-5 (9 AM weekdays), */15 * * * * (every 15 min), and 0 0 1 * * (start of each month).
